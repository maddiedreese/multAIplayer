# Compatibility and migration inventory

This is the repository-wide inventory of code that intentionally reads an older persisted, wire, or upstream shape. Product security claims remain in the [threat model](threat-model.md). A row may be removed only when its removal condition is met and the associated test is changed in the same pull request.

“Reject guard” means the old format is recognized only to produce a safe error; it is not accepted as current data. Additive normalizers may fill documented non-secret defaults, but no reader may silently accept an obsolete cryptographic wire format.

## Representation rule

`tsconfig.base.json` enables `strict`, `noUncheckedIndexedAccess`, and `exactOptionalPropertyTypes`. Wire and persistence normalizers omit absent properties; `null` is used only for an explicit protocol value. A decoded `undefined` must be reconstructed away before serialization. Purely in-memory display/callback types may use `optional?: T | undefined`, but those types must not be reused as persistence or wire contracts.

When adding a compatibility path, add its row, a bounded fixture or test, and a concrete removal condition in the same change. When removing one, include the search/test evidence that the condition has been met.
