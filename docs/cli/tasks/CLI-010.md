# CLI-010: Desktop release isolation and CI classification

State: waiting_for_owner_approval

## Outcome

Introduce explicit CLI/shared/desktop change classification and prove unfinished
CLI work cannot enter desktop packaging, updater, signing, version synchronization,
or required release assets.

## Allowed scope

- CLI-specific CI helpers and focused workflow classification.
- Documentation of release boundaries.
- Existing release files only after the owner approves each exact edit.

## Protected scope

- Runtime desktop behavior and dependency versions.
- Desktop artifact contract unless an explicit compatibility-only assertion is added.

## Acceptance criteria

- [ ] CLI-only changes run CLI checks without changing desktop artifacts.
- [ ] Shared changes require both client suites.
- [ ] Protected release paths are visibly reported or blocked.
- [ ] Existing desktop release verification remains green and semantically unchanged.

## Verification

- Existing release-tool tests and `npm run verify`.
- Path-classification fixtures for CLI, desktop, relay, and shared changes.
