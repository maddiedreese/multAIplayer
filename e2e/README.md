# End-to-end coverage

Protocol v2 keeps MLS private state in the native Rust core. Its end-to-end evidence is intentionally split into three layers so a UI simulation cannot be mistaken for cryptographic or process-integration coverage.

| Layer                                  | What is real                                                                                                                                                                                                                                  | Intentional boundary                                                                                                                                                                                     | CI gate            |
| -------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------ |
| Chromium UI contracts                  | Production React components, invite codec and parser, and Codex context builders                                                                                                                                                              | The test-only harness simulates relay delivery, native MLS, persistence, and native Codex execution. Every spec asserts the visible `E2E coverage boundary` disclosure.                                  | `web-shell-e2e`    |
| Stateful native-core/relay composition | Two independent long-lived `mls-core` processes, real device signatures, a real relay and SQLite store, the real KeyPackage validator, WebSocket-delivered MLS application and handoff messages, old-host rejection, and successor continuity | This is an ephemeral `mls-core` process fixture. It does not call Tauri commands, Keychain-backed invite receipts, GitHub OAuth, or the desktop UI.                                                      | `web-and-relay`    |
| Two real Tauri shells                  | Two isolated Linux Tauri application processes and credential stores, production React/Tauri command paths, real native MLS/HPKE/Commit/Welcome processing, a real relay, SQLite, the real KeyPackage validator, and real WebSocket transport | Relay debug authentication substitutes only for GitHub OAuth. The fixture starts with both GitHub identities already on one team, so it proves device MLS admission—not GitHub OAuth or team invitation. | `native-shell-e2e` |

The Linux native-shell journey in [`native-shell/journey.ts`](native-shell/journey.ts) creates a room in the host shell; proves the executable validator rejects a bit-tampered native KeyPackage; denies one production invite and verifies the guest has no native MLS group; and rejects an expired capability before KeyPackage publication. For successful admission, it hard-kills the requesting Tauri process after request persistence, approves while that guest is offline, then deterministically restarts the real relay over the same SQLite store after Commit acceptance but before Welcome delivery. The same guest profile and credential store restart, recover the Welcome and native roster, and decrypt the next message. The journey then transfers host authority, verifies the relay's active-host record, and sends successor traffic. `tauri-driver` and WebKitWebDriver provide UI automation; they do not replace application code. A loopback proxy holds one real Welcome request across the relay restart but fabricates no response or state. Expiry is induced by an explicitly enabled loopback-only test control that backdates the real relay record; the production lookup, pruning, unchanged package count, and locked native state remain the asserted behavior.

The stateful core/relay journey in [`apps/relay/test/live-native-relay-journey.test.ts`](../apps/relay/test/live-native-relay-journey.test.ts) is deliberately narrower. It makes relay-delivered bytes drive two live engines and catches sequencing errors that a precomputed transcript cannot, while leaving production invite-command and credential-store claims to the Tauri journey. The deterministic process journey in [`apps/relay/test/process-security-journey.test.ts`](../apps/relay/test/process-security-journey.test.ts) separately retains the relay database/WAL/wire plaintext scans and removed-member exclusion evidence.

The native-shell job is a Linux compatibility test; macOS remains the supported release target. The pinned `tauri-driver` has no macOS backend, so CI does not claim that the macOS WKWebView UI is automated. The macOS packaging job instead reruns the real native-core/relay/validator composition journey on macOS. Run Chromium UI contracts with `npm run test:e2e`, the restart-proxy harness contracts with `npm run test:harness`, the stateful process case through the relay test suite, and the Linux native journey with `xvfb-run -a npm run test:e2e:native` after installing the CI-listed WebKit, Secret Service, and `tauri-driver` dependencies.

The native journey writes total and per-stage timings to `reports/native-shell-e2e/duration.json`. CI uploads the report, includes its table in the job summary, and warns after a run crosses six minutes. A separate blocking policy rejects totals above eight minutes, stages above their reviewed ceiling, and unreviewed stage-sequence changes; this preserves useful warning headroom while preventing wait or startup regressions from silently consuming the 30-minute recovery timeout.
