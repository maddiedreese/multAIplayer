# End-to-end coverage

Protocol v2 keeps MLS private state in the native Rust core. Its end-to-end evidence is intentionally split into four layers so a UI simulation or boot smoke cannot be mistaken for cryptographic or full journey coverage.

| Layer                                  | What is real                                                                                                                                                                                                                                        | Intentional boundary                                                                                                                                                                                                          | CI gate            |
| -------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------ |
| Chromium UI contracts                  | Production React components, onboarding state/components, invite codec and parser, Codex context builders, shared chat code/image rendering, typed activity disclosures, and raw-reasoning setting UI                                               | The test-only harness simulates relay/authentication, native account/folder/Codex operations, native MLS, persistence, and native Codex execution. Every spec asserts the visible `UI contract coverage boundary` disclosure. | `ui-contract`      |
| Stateful native-core/relay composition | Two independent long-lived `mls-core` processes, real device signatures, a real relay and SQLite store, the real KeyPackage validator, WebSocket-delivered MLS application and handoff messages, old-host rejection, and successor continuity       | This is an ephemeral `mls-core` process fixture. It does not call Tauri commands, Keychain-backed invite receipts, GitHub OAuth, or the desktop UI.                                                                           | `web-and-relay`    |
| Two real Tauri shells                  | Two isolated Linux Tauri application processes and credential stores, production onboarding create/join handlers, real native MLS/HPKE/Commit/Welcome processing, a real relay, SQLite, the real KeyPackage validator, and real WebSocket transport | Relay debug authentication substitutes for GitHub OAuth, host ChatGPT readiness uses an explicit test transition, and the project path is typed. The guest is not pre-enrolled in the newly created team.                     | `native-shell-e2e` |
| Packaged macOS WKWebView smoke         | A test-only packaged app in a real WKWebView, a visible Profile click and Account assertion, a real `app_version` call, and an open/position/close cycle for the native room-browser child WebView                                                  | The embedded driver is test-only. This proves boot, one visible interaction, and the native browser lifecycle, not the two-client MLS journey or input-level coverage of the whole app.                                       | `macos-desktop`    |

The Linux native-shell journey in [`native-shell/journey.ts`](native-shell/journey.ts) creates a room in the host shell; proves the executable validator rejects a bit-tampered native KeyPackage; denies one production invite and verifies the guest has no native MLS group; and rejects an expired capability before KeyPackage publication. For successful admission, it hard-kills the requesting Tauri process after its native encrypted store has committed the exact request, approves while that guest is offline, then deterministically restarts the real relay over the same SQLite store after Commit acceptance but before Welcome delivery. The host re-establishes the relay's intentionally process-local device session by signing a fresh challenge and retries the exact durable approval; it validates and reuses the staged Welcome rather than constructing another Commit. The same guest profile and credential store restart, replay the byte-identical sealed request without exposing the bearer capability to the webview, recover the Welcome and native roster, and decrypt the next message. The journey then transfers host authority, verifies the relay's active-host record, and sends successor traffic. `tauri-driver` and WebKitWebDriver provide UI automation; they do not replace application code. A loopback proxy holds one real Welcome request across the relay restart but fabricates no response or state. Expiry is induced by an explicitly enabled loopback-only test control that backdates the real relay record; the production lookup, pruning, unchanged package count, and locked native state remain the asserted behavior.

The stateful core/relay journey in [`apps/relay/test/live-native-relay-journey.test.ts`](../apps/relay/test/live-native-relay-journey.test.ts) is deliberately narrower. It makes relay-delivered bytes drive two live engines and catches sequencing errors that a precomputed transcript cannot, while leaving production invite-command and credential-store claims to the Tauri journey. The deterministic process journey in [`apps/relay/test/process-security-journey.test.ts`](../apps/relay/test/process-security-journey.test.ts) separately retains the relay database/WAL/wire plaintext scans and removed-member exclusion evidence.

The native-shell job is a Linux compatibility test; macOS remains the supported release target. Direct `tauri-driver` 2.0.6 supports Linux WebKitWebDriver and Windows EdgeDriver but still has no direct macOS backend. The macOS job uses a distinct pinned WebdriverIO embedded-driver path for the narrow packaged WKWebView smoke, then rebuilds without that test-only driver before production packaging. Run that smoke with `npm run test:e2e:native:macos-smoke` after building the test-only native app as CI does.

WebKitGTK on Linux and WKWebView on macOS are different ports of the WebKit engine family. That makes Linux frontend layout and UI-contract evidence more transferable to macOS than a Chromium fallback would be, but it does not transfer four platform boundaries: `WKScriptMessageHandler` versus the GTK port's IPC mechanism, Keychain versus Secret Service, window/process lifecycle, or code-signing and entitlement effects on spawned validator and relay processes. The macOS composition and packaging steps separately exercise process spawning, the child stdio protocol, validator invocation, relay networking, and bundle assembly. The embedded-driver smoke adds real WKWebView boot, visible-control, and IPC-handshake evidence; it is not a macOS two-shell journey.

Run Chromium UI contracts with `npm run test:ui-contract`, the stateful process cases through the relay test suite, and the Linux native journey with `xvfb-run -a npm run test:e2e:native` after installing the CI-listed WebKit, Secret Service, and `tauri-driver` dependencies. The native-shell helper tests are available directly with `node --import tsx --test e2e/native-shell/*.test.ts` when changing that harness; they are not a second merge gate over the real journey.

The Codex chat contracts in [`codex-chat-parity.spec.ts`](codex-chat-parity.spec.ts) and [`raw-reasoning-setting.spec.ts`](raw-reasoning-setting.spec.ts) exercise the production chat and model-settings components. They verify human and Codex code blocks, a validated generated raster attachment, nested reasoning/raw-reasoning disclosures, edits, tools, subagent spawning, and the off-by-default room-sharing control and warning. They are rendering and interaction evidence; native projector tests separately verify that raw content is suppressed unless the setting is enabled, and the app-server/image/blob paths retain their separately stated boundaries.

`npm run docs:screenshots` renders the README's full desktop workspace plus focused chat, browser, and terminal sections from harness scenarios that mount the production dark-mode shell and components. Every capture is bounded by its rendered section rather than a padded normalization canvas. The 1200×700 workspace and 760-pixel-wide focused sections are sized for legibility at their actual README display widths. The focused browser has no loaded preview and the focused terminal has no session output, so stale demo content cannot become part of the product imagery; onboarding is excluded from every capture. Edit the scenario/component source and regenerate the assets instead of replacing them with manually cropped website or app screenshots. The capture script asserts dimensions and content boundaries and rejects overflow.

The six-case onboarding contract in [`onboarding.spec.ts`](onboarding.spec.ts) renders the production assistant, checklist, Help recovery, and first-turn guide. It verifies equal keyboard-accessible create/join entry, persisted Explore/checklist state, Help reopen/reset, direct readiness repair, partial team/room retry without duplication, safe-default disclosure, blocking host-device verification without optimistic unlock, and draft-only starter prompts. Its visible boundary banner labels relay workspace operations, GitHub/ChatGPT authentication, native Codex/account/folder behavior, MLS verification/host approval, encrypted persistence, and Codex execution as simulated. Desktop unit tests separately cover the versioned storage allowlist, production adapter delegation, and fixed error projection.

The Linux two-shell journey now starts from focused clean-profile Welcome surfaces. The host selects create, verifies relay/GitHub readiness, advances past live third-party ChatGPT authorization through an explicit test transition, submits the production workspace/room form, accepts safe defaults, and enters the room. The guest selects join, proves local Codex is nonblocking, and submits the canonical HTTPS invitation through the production onboarding form before the real approval/MLS journey continues. GitHub identity still comes from the loopback-only relay debug fixture and the project path is typed rather than selected through the platform picker, so this is production-handler and native integration evidence—not live-provider or input-level folder-picker evidence. The packaged macOS smoke still asserts Welcome and chooses Explore; it is not macOS create/join evidence.

The native journey writes total and per-stage timings to `reports/native-shell-e2e/duration.json`. CI uploads the report, includes its table in the job summary, and warns after a run crosses six minutes. Those timings are operational evidence for diagnosing regressions, not a checked-in contract for today's stage sequence or shared-runner duration; the 30-minute timeout remains the fail-safe for a hung recovery journey.
